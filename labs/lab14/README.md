# Лабораторна робота №14 — Stateful Flow Orchestration (Ukrainian News)

### 1. Опис Use Case
Метою роботи є перехід від простого виклику інструментів (Lab 12) до керованого конвеєра зі збереженням стану (**Stateful Flow**) для аналізу та структурування українських новин. 
Система (News Analyst Assistant) приймає короткі тексти, автоматично фільтрує нерелевантний контент (прогноз погоди, клікбейт), а цільові новини передає в LLM для визначення категорії та вилучення ключових іменованих сутностей (локації, особи, організації). Результат завжди експортується у стабільній JSON-структурі.

### 2. Етапи Оркестрації (Flow Steps)
Пайплайн працює з єдиним мутабельним об'єктом стану (`FlowState`) і проходить через 6 обов'язкових кроків:
1. **Ingest:** Приймання сирого тексту, нормалізація (видалення зайвих пробілів/символів) та генерація унікального `case_id`. Відкидання порожніх запитів на ранньому етапі.
2. **Route:** Детермінована маршрутизація на основі списку ключових слів (Early Exit). Якщо текст містить маркери погоди або спаму, він маркується як `weather_skip` або `spam_skip` і йде в обхід LLM, суттєво економлячи токени та час обчислень.
3. **Execute:** Для цільових новин викликається локальна LLM (`Llama-3-8B-Instruct`), яка витягує дані суворо за заданою JSON-схемою.
4. **Validate:** Перевірка згенерованої структури через бібліотеку `jsonschema` (наявність категорії та трьох масивів сутностей).
5. **Fallback:** Безпечне відновлення (Safe Failure). Якщо LLM повертає зламаний синтаксис (наприклад, через Prompt Injection), запускається резервний сценарій: призначається категорія "General" і порожні масиви сутностей, а запис маркується прапорцем `needs_manual_review`.
6. **Export:** Формування фінального, передбачуваного JSON-контракту для збереження в базу даних (навіть якщо конвеєр упав на якомусь етапі).

### 3. Технічний стек та Модель
* **LLM Двигун:** Локальна модель `unsloth/llama-3-8b-Instruct-bnb-4bit` (4-бітне квантування для запуску на Colab GPU).
* **Валідація:** Бібліотека `jsonschema` для перевірки типів даних та обов'язкових полів.
* **Логування:** Модуль `src/flow_logger.py` для запису повного стану виконання кожного кроку у файл `flow_logs_lab14.jsonl`.

### 4. Метрики та Результати (на 10 Test Cases)
* **Flow completion rate:** ~90% (кейси, які успішно структурувалися або були свідомо та безпечно відхилені роутером).
* **Export valid rate:** 90-100% (зовнішні системи гарантовано отримують стабільний JSON-об'єкт із необхідними полями статусу).
* **Safe Failure Rate:** 100% перехоплення критичних збоїв (симуляція ін'єкцій) без падіння всього скрипта.

### 5. Головний інженерний висновок
Впровадження архітектури Stateful Flow вирішило головну проблему попередньої лабораторної роботи (Lab 12) — **Unnecessary Tool Calls (Зайві виклики)**. Завдяки детермінованому роутингу, прогнози погоди та клікбейт більше не навантажують LLM. Ізоляція стану в об'єкті `FlowState` зробила код надійним, а етапи валідації та fallback гарантують, що в підсумкову базу даних ніколи не потраплять синтаксичні помилки.
