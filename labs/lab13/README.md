# Lab 13: Multi-Agent Extraction System (UA Customer Reviews)

## 1. Який use case обрано
Екстракція структурованих метаданих (формат JSON) з україномовних відгуків та скарг клієнтів (UA Datasets). Ціль: автоматично витягувати назву компанії, категорію проблеми, сентимент (позитивний/негативний/нейтральний) та суму фінансових втрат у гривнях для подальшої аналітики.

## 2. Які агенти є в crew
Система складається з 4 спеціалізованих агентів:
1. **Triager** (Класифікатор/Маршрутизатор)
2. **Extractor** (Парсер)
3. **QA Reviewer** (Контролер якості)
4. **Repair Agent** (Ремонтник)

## 3. Який workflow
Лінійний пайплайн з циклом виправлення (Repair Loop):
`Input Text` ➔ `Triager` ➔ `Extractor` ➔ `QA Reviewer` ➔ `[Repair Loop]` ➔ `Final JSON / Fallback`.

## 4. Які delegation rules
*   Якщо текст є спамом або нерелевантним (визначає Triager) ➔ пропускаємо Extractor, йдемо у Fallback.
*   Extractor генерує JSON ➔ обов'язково передає його QA Reviewer-у.
*   Якщо Reviewer дає `accept` ➔ зберігаємо результат.
*   Якщо Reviewer дає `repair_needed` ➔ делегуємо Repair Agent-у.
*   Якщо Reviewer знаходить критичні суперечності або Repair не справляється ➔ делегуємо у Fallback (Manual Review).

## 5. Як працює Reviewer
QA Reviewer не парсить текст з нуля. Він бере готовий JSON від Extractor-а і порівнює його з оригінальним текстом. Він шукає:
*   Галюцинації (вигадані бренди або категорії, яких немає в тексті).
*   Невідповідність типів даних (наприклад, текст "півтори тисячі" або float `120.50` замість integer для поля `monetary_loss_uah`).
*   Логічні суперечності (неправильно визначений сентимент через сарказм).

## 6. Як працює fallback
Впроваджено концепцію **Safe Failure**. Якщо система стикається з відвертим спамом, неможливістю привести дані до правильного типу або логічними конфліктами, вона не падає з Exception (краш скрипта). Замість цього вона створює Fallback-відповідь: повертає всі дані, які вдалося безпечно витягти (`partial_output`), і встановлює прапорець `needs_manual_review: true`.

## 10. Головний висновок
Multi-Agent підхід є критично необхідним для роботи з емоційними та зашумленими даними, якими є відгуки користувачів. Single-Agent схильний до галюцинацій та "зламів" через порушення типів даних. Ізоляція ролей у Crew-архітектурі дозволяє парсити текст, жорстко перевіряти результат, точково ремонтувати його та м'яко відхиляти неякісні дані (Safe Failure), що гарантує чистоту фінальної бази даних.
