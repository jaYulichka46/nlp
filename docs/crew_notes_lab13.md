# Crew Notes: Multi-Agent Extraction System (Lab 13 - UA Datasets)

## 1. Який use case обрано
Екстракція структурованих даних (JSON) з україномовних відгуків та скарг клієнтів (Customer Reviews). Мета: автоматично отримувати назву компанії, категорію проблеми, сентимент тексту та суму фінансових втрат для аналітики якості сервісу.

## 2. Які агенти є в crew
У системі працює 4 агенти:
1. **Triager** (Маршрутизатор)
2. **Extractor** (Парсер)
3. **QA Reviewer** (Контролер якості)
4. **Repair Agent** (Ремонтник)

## 3. Роль кожного агента
*   **Triager:** Аналізує текст і визначає, чи це дійсно відгук/скарга, чи просто спам/інший текст. Визначає складність.
*   **Extractor:** Витягує цільові поля у форматі JSON згідно з жорсткою схемою (без перевірок).
*   **QA Reviewer:** Порівнює згенерований Extractor-ом JSON із початковим текстом. Шукає вигадані факти, невідповідності типів даних та логічні суперечності.
*   **Repair Agent:** Отримує зауваження від Reviewer-а і намагається точково виправити зламаний JSON (наприклад, перетворити float на int або видалити вигаданий бренд).

## 4. Правила делегування
*   **Вхід:** Текст іде до Triager-а.
*   **Triager -> Extractor:** Якщо текст розпізнано як релевантний. Якщо це спам — миттєвий Fallback.
*   **Extractor -> Reviewer:** Сформований JSON передається на перевірку.
*   **Reviewer -> Final:** Якщо все ідеально (verdict `accept`).
*   **Reviewer -> Repair:** Якщо є дрібні помилки типу або галюцинації (verdict `repair_needed`).
*   **Reviewer -> Fallback:** Якщо знайдено суперечності сентименту або текст надто розмитий (verdict `manual_review`).

## 5. Що перевіряє Reviewer
*   **Галюцинації (Hallucinations):** чи не вигадав Extractor назву компанії або проблему, яких немає в короткому емоційному відгуку.
*   **Типи даних (Type Mismatch):** чи записані фінансові втрати (`monetary_loss_uah`) як ціле число (integer), а не рядок чи дріб.
*   **Консистентність (Consistency):** чи збігається встановлений сентимент із реальним тоном тексту (особливо важливо для змішаних відгуків).

## 6. Коли спрацьовує fallback
Fallback (Safe Failure) активується, коли:
1. Triager маркує текст як нерелевантний (наприклад, оголошення про продаж гаража).
2. Reviewer бачить повні логічні суперечності (сарказм, "все супер, але гроші вкрали").
3. Repair Agent не зміг виправити критичну помилку схеми після підказки.
У цих випадках система не падає, а повертає статус `needs_manual_review`.

## 7. Що crew покращив порівняно з single-agent
*   **Фільтрація сміття:** Single-agent намагався знайти компанію та проблему навіть у спамі. Crew одразу це відсікає.
*   **Відсутність галюцинацій:** У коротких відгуках типу "Сервіс дно" Extractor часто вигадував бренд. Reviewer успішно це припиняє.
*   **Стабільність типів:** Вирішено проблему конвертації сум прописом у строгий integer (завдяки Repair).

## 8. Де multi-agent підхід був зайвий
На ідеально структурованих та однозначних відгуках (наприклад: "Замовляв телефон у Розетці. Доставили швидко, сервіс супер"). Тут Extractor справляється з першого разу, а додаткові агенти лише витрачають токени та час API.

## 9. Які помилки залишилися
*   Висока чутливість до неявного сарказму (система може сприйняти "Дякую Новій Пошті за розбитий телевізор" як positive sentiment, якщо Reviewer не вловить контекст).
*   Складнощі з обчисленням математики в тексті ("списали двічі по 400 грн").

## 10. Що б ви фіксили далі
*   **Few-Shot для сарказму:** Додав би у системні промпти Reviewer-а кілька прикладів типового українського сарказму у відгуках.
*   **Оптимізація викликів:** Зробив би так, щоб на дуже простих текстах (за оцінкою Triager-а) Reviewer пропускався взагалі, економлячи ліміти API.
