# Agent Notes Lab 12: News Analyst Assistant

**1. Який use case обрано**
Варіант 1: Support / News Analyst Assistant. Аналіз коротких текстів українських новин для структурування медіа-потоку.

**2. Який agent task**
Отримання тексту новини та автоматичне вилучення двох типів інформації: категорії новини (War, Politics, Economy тощо) та ключових сутностей (локації, люди, організації).

**3. Які tools реалізовано**
* `categorize_news`: визначає категорію новини на основі підрахунку ключових слів-маркерів.
* `extract_key_entities`: детермінований пошук локацій, організацій та відомих осіб за допомогою заздалегідь підготовленого rule-based словника.

**4. Як агент вирішує, коли викликати tool**
Агент працює за парадигмою ReAct (Reason + Act). У системному промпті йому надано описи інструментів та жорстка вимога повертати відповідь у форматі JSON. Отримуючи новину, LLM генерує поле `thought`, де розуміє необхідність класифікувати текст та знайти сутності. Агент формує виклик `tool_name` з текстом новини, отримує від оркестратора JSON із результатом роботи функції, аналізує його і формує фінальну структуровану відповідь.

**5. Як логуються tool calls**
Реалізовано окремий клас `ToolLogger`. Кожен виклик інструменту автоматично перехоплюється і записується у файл `docs/tool_logs_lab12.jsonl` у форматі JSON Lines. Для кожного виклику фіксуються: `timestamp`, `task_id`, `tool_name`, `input`, `output`, `success` (boolean) та можливі помилки.

**6. Що tools реально покращили**
Інструменти розв'язали проблему "творчості" LLM. Базова LLM (Baseline) часто вигадувала власні категорії новин, що робило б неможливим побудову аналітичних дашбордів. Інструмент `categorize_news` жорстко обмежив вибір, гарантуючи передбачувану категоризацію.

**7. Де tools були зайві або не допомогли**
Виклики інструменту екстракції сутностей були зайвими для прогнозу погоди та клікбейтних новин (Unnecessary tool calls). Агент марно витрачав час на пошук політиків у новині про температуру повітря. Також інструмент не допоміг із спортивними новинами, оскільки відповідних сутностей не було у хардкод-словнику.

**8. Які помилки лишилися**
* **Сліпота до нових сутностей:** Оскільки `extract_key_entities` базується на словнику, він пропускає людей або локації, які не були внесені в базу розробником (наприклад, нових міністрів або невеликі міста).
* **Відсутність раннього переривання:** Агент викликає всі доступні інструменти, навіть якщо перший інструмент визначив новину як нерелевантну.

**9. Що б ви фіксили далі**
1. **Інтеграція справжньої NER-моделі:** Замінити примітивний словниковий пошук на виклик моделі SpaCy (натренованої на українських текстах), щоб знаходити сутності зважаючи на контекст, а не лише за списком.
2. **Впровадження Chain of Thought:** Навчити агента спочатку викликати класифікатор, і якщо новина належить до категорії "General/Unknown", відразу завершувати роботу без виклику пошуку сутностей.
